# PRD: Personal Task Tracker

## Overview

Минималистичный персональный таск-трекер в стиле заметок. Без команд, без ролей, без лишних функций. Один пользователь, один экран, всё на виду.

Продукт решает проблему: существующие трекеры (Jira, Linear, Notion) избыточны для соло-разработчика/продакта. Нужен инструмент на уровне сложности текстового файла, но со структурой и персистентностью.

---

## Сущности

### Epic

Крупная цель или направление работы. Не имеет собственного описания — только название. Служит группирующим тегом для задач.

Поля:
- `id` — уникальный идентификатор
- `text` — название эпика

Поведение:
- Создаётся inline в списке эпиков
- Редактируется кликом по названию
- Имеет карточку (модалку) с названием и списком всех привязанных задач
- Привязка двусторонняя: линк таски к эпику автоматически отображает таску в карточке эпика
- Счётчик привязанных задач виден рядом с названием в списке
- Из карточки эпика можно перейти в карточку любой привязанной таски
- Архивируется отдельной кнопкой ✓ → попадает в секцию "Epics" архива
- Удаляется кнопкой × (без архивации, безвозвратно)
- Связи epic↔task сохраняются при архивации обеих сущностей

### Task

Единица работы. Всегда принадлежит спринту.

Поля:
- `id` — уникальный идентификатор
- `text` — название задачи
- `desc` — описание (свободный текст)
- `epicId` — ссылка на эпик (nullable)
- `done` — булево состояние выполнения
- `createdAt` — timestamp создания

Поведение:
- Создаётся кнопкой "+ add" внутри спринта
- Название редактируется inline
- Имеет состояние done/undone, переключаемое кнопкой ○/✓
  - done-таска: зачёркнутый текст, сниженная непрозрачность, зелёная галочка
  - Остаётся в спринте (не перемещается автоматически)
- Имеет карточку (модалку) по кнопке ≡:
  - Название (read-only в карточке, редактируется inline)
  - Дата и время создания
  - Выбор эпика из dropdown (все активные + архивные эпики)
  - Полноэкранное поле описания
  - Кнопки cancel / save
- Перемещается между спринтами кнопкой → с dropdown-выбором целевого спринта
- Удаляется кнопкой × (безвозвратно)
- При архивации спринта — архивируется вместе с ним, сохраняя структуру
- Рядом с названием отображается `#epicName` если привязана к эпику

### Sprint

Временной контейнер для задач. Аналог недельного/двухнедельного цикла.

Поля:
- `id` — уникальный идентификатор
- `name` — название (по умолчанию "Week N", редактируемое)
- `goal` — цель спринта (короткая строка)
- `desc` — описание/контекст спринта (длинный текст)
- `tasks` — массив задач

Поведение:
- Создаётся кнопкой "+ add sprint" с автоинкрементом номера
- Название редактируется inline
- Goal отображается под названием в основном списке
- Имеет карточку (модалку) по кнопке ≡:
  - Name (input)
  - Goal (input)
  - Description (textarea)
  - Кнопки cancel / save
- Счётчик `done/total` задач в заголовке
- Архивируется целиком кнопкой "✓ archive" → все задачи сохраняют структуру в архиве
- Удаляется кнопкой × (безвозвратно, вместе со всеми задачами)

## Архив

Три раздельные секции, скрытые по умолчанию за кнопкой "Archive ▸" с общим счётчиком.

### Секция "Epics"
- Список архивированных эпиков
- Каждый: кнопка ↩ (restore), название (кликабельно → карточка, read-only), кнопка × (удалить)
- Restore → возвращает в активные эпики

### Секция "Sprints"
- Список архивированных спринтов в компактном виде (свёрнуты)
- Заголовок: ↩, ▸/▾ (раскрыть), название, goal, количество задач, дата архивации, ×
- Раскрытый спринт показывает все задачи с состоянием done/undone и привязкой к эпику
- Клик по задаче → карточка задачи (read-only)
- Restore → возвращает спринт со всеми задачами в активные

### Секция "Tasks"
- Отдельные задачи, архивированные поштучно (не в составе спринта)
- Каждая: ↩, название (кликабельно → карточка, read-only), эпик, дата архивации, ×
- Restore → помещает в первый активный спринт. Если спринтов нет — создаёт "Backlog"

### Сохранение связей
- Архивированный эпик продолжает отображаться в dropdown при редактировании задачи
- Задачи, привязанные к архивному эпику, сохраняют привязку
- Карточка архивного эпика показывает все привязанные задачи (активные и архивные) с указанием местоположения

## Операции между сущностями

### Перемещение задач (→)
- Из спринта A в спринт B
- Dropdown показывает все спринты кроме текущего
- Задача сохраняет все поля (desc, epicId, done, createdAt)

### Двусторонние связи epic↔task
- Привязка задаётся в карточке задачи (dropdown эпиков)
- В карточке эпика автоматически отображается список всех привязанных задач
- Задачи в карточке эпика показывают: название, местоположение (спринт/архив), статус done
- Клик по задаче в карточке эпика → открывает карточку задачи

## UI/UX

### Общие принципы
- Моноширинный шрифт (SF Mono → Menlo → Consolas)
- Максимальная ширина контента: 560px, центрирован
- Нет декоративных элементов, теней, градиентов в основном интерфейсе
- Цветовая палитра: #222 (основной текст), #888-#aaa (вторичный), #ccc-#ddd (UI-элементы), #22c55e (done/archive)
- Inline-редактирование: клик по тексту → input, Enter/blur → сохранение, Escape → отмена

### Иерархия элементов на экране (сверху вниз)
1. Epics — список эпиков с кнопками ≡ ✓ ×
2. Спринты (в порядке создания) — каждый с заголовком, goal, задачами
3. "+ add sprint"
4. Archive (свёрнут)
5. Reset

### Модальные окна
- Оверлей rgba(0,0,0,0.3), клик по оверлею закрывает
- Белый блок 460px max, border-radius 8, тень
- Кнопка × в правом верхнем углу
- Cancel/Save в правом нижнем
- Read-only режим для архивных элементов (disabled inputs, нет кнопки save)

### Интерактивные элементы задачи (слева направо)
1. ○/✓ — toggle done
2. – — визуальный маркер списка
3. Название (inline-editable) + #epicName
4. → — move dropdown
5. ≡ — открыть карточку
6. × — удалить

## Хранение данных

- Персистентный key-value storage (`window.storage`)
- Один ключ `tracker-v5` — всё состояние в одном JSON
- Автосохранение при каждом изменении
- Кнопка "reset" — сброс до пустого состояния (с confirm)

### Схема данных

```json
{
  "epics": [
    { "id": "string", "text": "string" }
  ],
  "sprints": [
    {
      "id": "string",
      "name": "string",
      "goal": "string",
      "desc": "string",
      "tasks": [
        {
          "id": "string",
          "text": "string",
          "desc": "string",
          "epicId": "string | null",
          "done": "boolean",
          "createdAt": "number (timestamp)"
        }
      ]
    }
  ],
  "archiveTasks": [
    { "...task fields", "archivedAt": "number" }
  ],
  "archiveSprints": [
    { "...sprint fields", "archivedAt": "number" }
  ],
  "archiveEpics": [
    { "...epic fields", "archivedAt": "number" }
  ],
  "sc": "number (sprint counter)"
}
```

## Что сознательно исключено

- Даты дедлайнов и календарь
- Приоритеты и лейблы
- Мультипользовательский режим
- Поиск и фильтры
- Drag-and-drop сортировка
- Уведомления
- История изменений / undo
- Статусы задач кроме done/undone
- Подзадачи
- Описание в эпиках

Каждое из этих решений — осознанный выбор в пользу простоты. Добавление любого из этих элементов должно пройти проверку: "стоит ли усложнение того value, которое оно даёт одному пользователю?"
